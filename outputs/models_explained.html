Part 1 — Foundation: Decision Trees

All three models (XGBoost, LightGBM, Extra Trees) are built on decision trees. Understanding a single tree is the key to understanding everything else.

What is a Decision Tree?

A decision tree is a flowchart-like structure. It asks a series of yes/no questions about the data, and each answer leads to a prediction.

Root Node: floor_area_sqm > 90? ├── Yes → Decision Node: dist_to_cbd < 5km? │ ├── Yes → Leaf: predict $650,000 │ └── No → Leaf: predict $480,000 └── No → Decision Node: remaining_lease > 70? ├── Yes → Leaf: predict $420,000 └── No → Leaf: predict $320,000
Node typeWhat it doesHDB example
Root NodeThe first split — most important feature goes herefloor_area_sqm > 90
Decision NodeHas 2+ branches — tests a featuredist_to_cbd < 5km
Leaf NodeFinal answer — no more splitspredict $480,000

How Does the Tree Decide Where to Split? (CART)

Sklearn uses CART (Classification and Regression Trees). It evaluates every possible split point on every feature and picks the one that reduces impurity the most.

Gini Impurity

Measures how often a randomly chosen element would be incorrectly labelled. Low Gini = pure node (all same class). Used by default in sklearn Decision Trees.

Like a bag of marbles — if all marbles are the same colour (pure), Gini = 0. If half are red and half blue, Gini = 0.5 (maximum disorder).

Entropy & Information Gain

Entropy measures randomness/disorder in a node. Information Gain = how much entropy drops after a split. The split with the highest information gain is chosen.

Like asking: "After I split here, am I more certain about the answer?" The split that gives the most certainty wins.

The Problem: A Single Tree Overfits

A single decision tree, if grown deep enough, will memorise the training data — it fits every quirk and noise in the dataset. On new data, it performs poorly.

Pros of Decision Trees: Simple, interpretable, handles both numerical and categorical data, requires little preprocessing.
Cons: Prone to overfitting with many features. Unstable — small data changes can produce a completely different tree.
Analogy: One doctor diagnoses you based on their personal experience. Sometimes brilliant, sometimes wrong. A panel of doctors with different specialities is more reliable — that's ensemble learning.

Part 2 — Two Ensemble Families: Bagging vs Boosting

Ensemble learning combines many weak models into one strong model. The syllabus covers two main approaches: Bagging and Boosting. Our three models each belong to one of these families.

BaggingBoosting
Full nameBootstrap AggregatingSequential Error Correction
How trees are builtIn parallel — each tree is independentIn sequence — each tree fixes the previous tree's errors
Data samplingEach tree trains on a random bootstrap sample (with replacement)Each tree trains on residuals/errors of the prior model
Final predictionAverage (regression) or vote (classification)Weighted sum of all trees
Main benefitReduces variance — prevents overfittingReduces bias — corrects systematic errors
Our modelsExtra TreesXGBoost LightGBM

Bagging ET

  1. Sample N subsets from training data (with replacement)
  2. Train one independent tree on each subset
  3. Average all predictions

Each tree sees different data → diverse errors → averaging cancels them out.

Boosting XGB LGBM

  1. Train Tree 1 on original data
  2. Calculate Tree 1's errors (residuals)
  3. Train Tree 2 to predict those errors
  4. Add Tree 2 to Tree 1. Repeat 1,500 times.

Each tree is a small correction to everything before it.

Analogy from class: Think of an orchestra (ensemble). Bagging = all musicians play simultaneously and we average the sound. Boosting = musicians take turns, each one listening to what the previous played and adding what was missing.

Where does Random Forest fit?

Random Forest (covered in the syllabus) is the direct predecessor of Extra Trees. Both are bagging methods. Random Forest = bagging + random feature subsampling per tree. Extra Trees goes one step further — it also randomises the split thresholds.

Part 3 — Extra Trees Bagging Family

What makes Extra Trees "Extra"?

Full name: Extremely Randomised Trees. It adds an extra layer of randomness on top of Random Forest.

Random ForestExtra Trees
Data samplingBootstrap (with replacement)Whole training set (no sampling)
Feature selection per splitRandom subset of featuresRandom subset of features
Split thresholdSearches for best thresholdPicks a random threshold
SpeedMediumFast (no threshold search)
VarianceLowLower (more randomness)

Why random thresholds?

Random Forest: "For feature dist_to_cbd, I'll try 0.5km, 1km, 2km, 5km… I'll pick 2km because that split reduces error the most."

Extra Trees: "For feature dist_to_cbd, I'll randomly pick 3.7km. Done."

Key insight: The randomness makes ET trees more diverse from each other. When you average 700 diverse trees, the random errors cancel out. The result is a model with very low variance — it doesn't overfit easily.

Our HDB Extra Trees Model (v8)

ExtraTreesRegressor(
    n_estimators    = 700,    # 700 independent trees
    max_features    = 0.8,    # Each split considers 80% of features randomly
    max_depth       = None,   # Trees grown until pure leaves (no depth cap)
    min_samples_split = 10,   # Node needs at least 10 samples to split further
    min_samples_leaf  = 1,
    random_state    = 42,
    n_jobs          = -1      # Use all CPU cores
)

Why is ET's individual RMSE higher ($23,478 vs XGB's $21,987)?
Random split thresholds are less precise than optimised ones. ET sacrifices some accuracy per tree in exchange for diversity. This trade-off is worth it when stacking — the diversity it adds helps Ridge correct errors the boosting models share.

Part 4 — XGBoost Gradient Boosting Family

Gradient Boosting — the Core Idea

The syllabus describes gradient boosting as: "a variant of boosting that uses gradient descent to minimise errors in sequential models. It builds one tree at a time, where each new tree helps to correct errors made by previously trained trees."

Round 1: Train Tree_1 → predicts $400K, actual $480K → residual = +$80K Round 2: Train Tree_2 on residuals → predicts +$60K Round 3: Train Tree_3 on new residuals → predicts +$15K ... Round 1500: Final prediction = sum of all 1500 small corrections

What XGBoost adds on top of plain Gradient Boosting

FeatureWhat it does
L1 regularisation (reg_alpha)Penalises the absolute values of leaf weights → can zero out useless features
L2 regularisation (reg_lambda)Penalises squared leaf weights → shrinks all weights, prevents any one feature from dominating
Tree pruningGrows trees then prunes back branches that don't improve the score
Missing value handlingLearns which branch to send missing values down automatically
Column subsampling (colsample_bytree)Each tree only sees a random fraction of features → reduces correlation between trees

Our HDB XGBoost Model (v7/v8)

XGBRegressor(
    n_estimators      = 1500,   # 1500 sequential trees
    max_depth         = 7,      # Each tree can be at most 7 levels deep
    learning_rate     = 0.05,   # Each tree contributes 5% of its prediction
    subsample         = 0.9,    # Each tree trains on 90% of rows (randomly)
    colsample_bytree  = 0.6,    # Each tree sees 60% of features
    min_child_weight  = 1,
    reg_alpha         = 0.01,   # Small L1 penalty
    reg_lambda        = 2,      # L2 penalty
    tree_method       = 'hist', # Fast histogram-based algorithm
    random_state      = 42,
    n_jobs            = -1
)

Analogy: XGBoost is like a very diligent student who reviews every exam mistake carefully and focuses on weak areas next time. After 1,500 rounds of correction, it's highly accurate — but it can sometimes over-study (overfit) without the regularisation penalty keeping it honest.

Part 5 — LightGBM Gradient Boosting Family

What is LightGBM?

From the syllabus: "LightGBM (Light Gradient Boosting Machine) is a gradient boosting framework developed by Microsoft that uses tree-based learning algorithms and is designed for distributed and efficient training, particularly on large datasets."

"Currently one of the best Supervised Learning algorithms in terms of performance and speed for tabular (structural or dataframe-like) datasets."

LightGBM's Key Innovation: Leaf-wise Growth

XGBoost and most gradient boosting methods use level-wise tree growth. LightGBM uses leaf-wise growth. This is the most important difference to understand.

Level-wise (XGBoost default)

Split all leaves at each depth level:

Depth 1: split 1 node → 2 nodes Depth 2: split 2 nodes → 4 nodes Depth 3: split 4 nodes → 8 nodes (balanced tree)

Predictable structure, less risk of overfitting.

Leaf-wise (LightGBM)

Always split the single leaf with the highest loss gain:

Step 1: split best leaf → 2 nodes Step 2: split best leaf → 3 nodes Step 3: split best leaf → 4 nodes (asymmetric, deep where needed)

Faster convergence, better accuracy per tree — but needs num_leaves to control depth.

Why this matters: Leaf-wise growth means LightGBM can fit complex patterns more efficiently. It gets better accuracy with fewer trees — hence "Light" (faster). The key control parameter is num_leaves, not max_depth alone.

LightGBM Pros & Cons (from syllabus)

Advantages

  • Faster training and higher efficiency
  • Lower memory usage
  • Better accuracy with parallel and GPU learning
  • Capable of handling large-scale data

Disadvantages

  • May overfit on small datasets
  • Requires careful hyperparameter tuning (especially num_leaves)

Our HDB LightGBM Model (v7/v8)

LGBMRegressor(

# ET coefficient 0.22 — clearly positive, meaningful contribution
ET's coefficient (0.22) is well above zero. Even though ET is the weakest individual model ($23,478), Ridge still uses it significantly. Why? Because ET makes different errors from XGB and LGBM. When both boosting models are wrong about the same flat, ET's independent (random) structure is sometimes right — and Ridge exploits that.

Why Do Bagging and Boosting Models Make Different Errors?

XGBoost & LightGBMExtra Trees
Error typeCorrelated — both struggle with same unusual flats (e.g. very large executive flats in non-mature estates)Random — errors are scattered, uncorrelated with boosting errors
When combinedShared errors survive averagingRandom errors cancel out; occasional "lucky" correct predictions add value
Analogy: Three specialists reviewing an HDB valuation. Surgeon (XGB) and GP (LGBM) both went to the same medical school — they tend to miss the same rare conditions. Nutritionist (ET) has a completely different training background — when both doctors are wrong about an unusual case, the nutritionist's different angle catches it.

Part 10 — Glossary

TermSimple Definition
Gini ImpurityHow often a randomly chosen element would be incorrectly labelled. 0 = perfectly pure node. Used to choose splits.
EntropyMeasure of disorder in a node. High entropy = mixed labels. Low entropy = mostly one class.
Information GainHow much entropy decreases after a split. We pick the split with the highest information gain.
BootstrappingSampling with replacement — draw N rows from N-row dataset, some rows appear twice, some not at all. Used in Random Forest and Bagging.
OOF PredictionOut-of-Fold — predicting a row using a model that was trained without that row. Prevents target leakage.
ResidualsThe errors from one model — what the next boosting tree tries to predict.
Leaf-wise growthLightGBM's approach — always split the single leaf with the highest gain. Creates asymmetric, efficient trees.
Level-wise growthTraditional approach — split all leaves at each depth. More balanced, less efficient.
L1 regularisation (Lasso)Penalty on absolute values of weights. Can shrink weights to zero — effectively removing features.
L2 regularisation (Ridge)Penalty on squared values of weights. Shrinks all weights towards zero but rarely to exactly zero.
subsampleFraction of training rows used per tree. Adds randomness, reduces overfitting.
colsample_bytreeFraction of features randomly selected per tree. Reduces correlation between trees.
num_leavesLightGBM's main complexity control. More leaves = more complex model. Use instead of max_depth for LGBM.
CV–Kaggle gapDifference between cross-validation RMSE and Kaggle public RMSE. Small gap = model generalises well.
Ridge meta-modelIn stacking: a simple linear model that learns the optimal weights to combine base model predictions.
