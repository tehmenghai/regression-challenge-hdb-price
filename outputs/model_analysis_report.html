2. Data Pipeline Steps

The end-to-end flow from raw CSV to Kaggle submission prediction:

Step 1: Raw data ingestion
   train.csv (150,634 × 77)  +  test.csv (16,737 × 76)

Step 2: Drop redundant columns
   floor_area_sqft (duplicate), lower/upper/mid (duplicate of mid_storey),
   full_flat_type, address, Tranc_YearMonth, residential, year_completed

Step 3: Fill missing values
   - Amenity count columns (e.g. Hawker_Within_500m)  → 0 (none present)
   - Numeric columns  → median imputation (inside sklearn Pipeline)
   - Categorical cols → most_frequent + OrdinalEncoder

Step 4: Feature engineering  (16 new features — see Section 3)

Step 5: OOF target encoding  (v6+)
   - town_median_price            (OOF, 5-fold)
   - postal_sector_median_price   (OOF, 5-fold)  [v9+]
   - flat_model_median_price      (OOF, 5-fold)  [v9+]

Step 6: Drop high-cardinality raw columns
   - postal (9,125 levels) → replaced by postal_sector_median_price
   - block  (2,514 levels) → replaced by block_num (numeric)

Step 7: Log-transform target  (v6+)
   y_log = log1p(resale_price)
   → reduces outsized influence of $1M+ flats on squared-error loss

Step 8: 5-Fold KFold split  (shuffle=True, random_state=42)

Step 9: Hyperparameter tuning  (RandomizedSearchCV, inner 3-fold CV)

Step 10: OOF prediction generation  (stacking, v7+)
   Each base model trained 5×; val predictions form OOF vector (n=150,634)

Step 11: Ridge meta-learner
   Trained on [xgb_oof, lgbm_oof, et_oof] → learns optimal blend weights

Step 12: Test predictions
   Base models averaged over 5 folds → Ridge → expm1() inversion

Step 13: Kaggle submission CSV
   Format: Id, Predicted

Why log1p the target? An RMSE of $30,000 on a $600K flat (5%) hurts the loss just as much as $30,000 on a $300K flat (10%). Log-transforming the target brings the prediction task closer to minimising percentage errors — reducing the disproportionate influence of expensive outliers.

Key insight: why engineered features help boosting but not RF. RF v1 scored $25,943 with raw features. Adding engineered features worsened RF to $27,582 (v2) because RF already implicitly captures lat/lon interactions through coordinate splits. 5. Validation Strategy

5.1 Cross-Validation Design

5-Fold KFold — KFold(n_splits=5, shuffle=True, random_state=42)

  • Dataset is randomly shuffled once, then split into 5 equal folds (~30,127 rows each)
  • Each fold serves as validation exactly once; the model trains on the remaining 4 folds
  • KFold (not StratifiedKFold) — regression target does not need stratification
  • random_state=42 ensures reproducibility across all versions

5.2 Out-of-Fold (OOF) Predictions

OOF predictions are the backbone of honest stacking:

For each fold i of 5:
   1. Train base model on rows NOT in fold i
   2. Predict rows IN fold i
   → store in oof_predictions[fold_i_indices]

After all 5 folds:
   oof_predictions.shape = (150,634,)
   Each row's prediction was made by a model that NEVER trained on it

This "never trained on the row being predicted" property makes OOF predictions honest — using them to train Ridge prevents the meta-model from seeing artificially easy (memorised) predictions.

5.3 OOF Target Encoding — Preventing Feature Leakage

Computing town_median_price = median(resale_price per town) on the full training set is leakage: each row's encoding partially reflects its own label. This inflated CV scores (v5 gap was $858).

OOF fix (v6+): For each of the 5 folds, the median map is computed from the other 4 folds only.

for tr_idx, va_idx in kf.split(groups):
    fold_map = {g: median(y[tr_idx] for group g)}
    encoded[va_idx] = fold_map[groups[va_idx]]  # val row's price excluded

Double OOF (v7+): Inside the stacking OOF loop, the three group encodings are re-computed from each fold's training rows — preventing leakage at the stacking stage too. This is why the v6→v7 CV–Kaggle gap collapsed from $306 → $65.

5.4 Hyperparameter Tuning Validation

RandomizedSearchCV uses an inner 3-fold CV inside each trial:

  • Outer loop: 5-fold KFold (model evaluation)
  • Inner loop: 3-fold CV inside RandomizedSearchCV (hyperparameter selection)
  • Search budget: 40 trials for XGB/LGBM (v9), 15 trials for ET

Custom scoring forces the search to directly optimise dollar-RMSE (not log-space RMSE):

def neg_dollar_rmse(y_log_true, y_log_pred):
    return -sqrt(MSE(expm1(y_log_true), expm1(y_log_pred)))

Improvement Inflection Points

1. v2 → v3: Biggest single gain (–$4,781)

Switching from Random Forest to tuned XGBoost. The same feature set that hurt RF in v2 ($27,582 vs $25,943) gave XGBoost a dramatic improvement in v3. Gradient boosting explicitly leverages engineered features through sequential error-correction; RF builds implicit interactions naturally from raw coordinates.

2. v5 → v6: Leakage fix (–$304, gap $858 → $306)

town_median_price was computed on the full training set in v5, causing target leakage that inflated CV scores while Kaggle scores lagged. OOF encoding and log1p() target transform in v6 fixed this: the gap collapsed from $858 to $306, confirming CV was genuinely misleading before.

3. v6 → v7: OOF stacking (–$218, gap $306 → $65)

Replacing fixed-ratio blending with Ridge meta-stacking on OOF predictions collapsed the CV–Kaggle gap to $65 — the best generalisation in the project. The meta-model learns adaptive weights rather than applying the same XGB/LGBM ratio across all predictions.

4. v7 → v8: Bagging diversity (–$101)

Adding ExtraTrees (random split thresholds, parallel tree construction) to the stack brought fundamentally different errors from the two boosting models. Ridge assigned it a 22% weight despite it being weaker individually ($23,587 OOF vs $21,841 stack). Model diversity is more valuable than raw model accuracy.

5. v8 → v9: Richer OOF encoding (–$49)

Replacing 9,125-level ordinal-encoded postal with 598-level postal_sector_median_price (OOF median) gave trees a clean geographic price signal. Block and flat model OOF encodings added further signal at finer granularity.

7. Key Learnings & Conclusions

Algorithm Choice Matters More Than Feature Engineering

The single largest improvement (–$4,781) came from switching RF → XGBoost, not from adding features. The same features that made RF worse (+$1,639 in v2) gave XGBoost a –$4,781 boost in v3. The lesson: match the model family to the feature representation. Explicit, linearised signals (dist_to_cbd, remaining_lease) benefit gradient boosting far more than Random Forest.

Target Leakage Is Measurable and Fixable

The $858 CV–Kaggle gap in v5 was a precise, quantifiable signal of target encoding leakage. After the OOF fix in v6, the gap fell to $306. This makes gap analysis a reliable diagnostic — when CV and Kaggle diverge significantly, investigate target-encoded features first.

OOF Stacking Generalises Better Than Fixed Blending

The Ridge meta-learner learns context-sensitive blend weights from OOF predictions rather than applying a fixed ratio. The gap collapse from $306 → $65 (v6 → v7) confirms the meta-model is not overfitting: it generalised immediately to unseen Kaggle data at near-perfect fidelity.

Model Diversity Beats Model Strength

ExtraTrees (individual OOF RMSE $23,587) earned a positive 0.1347 coefficient in the v9 Ridge meta-model. This confirms it adds genuine signal that XGBoost and LightGBM together do not. The diversity is architectural: bagging with random split thresholds vs sequential boosting with optimised thresholds.

Geographic Granularity Is High Value

The shift from 26-group town encoding to 598-group postal sector encoding was the single largest change in v9. Singapore's housing market is hyper-local — resale prices within one town can vary by $200K+ across postal sectors. Finer geographic features are among the highest-value improvements available.
