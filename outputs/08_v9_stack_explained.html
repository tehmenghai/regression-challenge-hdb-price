Feature Engineering — New Features

All v8 features carry forward unchanged. Three new features are added:

# 1. Postal sector: first 4 chars ("520123" → "5201")
df['postal_sector'] = df['postal'].astype(str).str[:4]

# 2. Block number: extract digits ("123A" → 123)
df['block_num'] = pd.to_numeric(
    df['block'].astype(str).str.extract(r'(\d+)')[0], errors='coerce'
).fillna(0)

postal and block (raw high-cardinality columns) are dropped entirely. postal_sector is used only to compute its OOF median price — the final model sees only the numeric postal_sector_median_price.

Why 4 characters? Singapore postal codes are 6 digits. The first 2 digits are the sector (28 sectors) and digits 3–4 are the district. Using 4 characters gives 598 unique groups — fine enough to distinguish neighbourhoods, coarse enough to have enough samples per group for a stable median.

Per-Fold OOF Target Encoding

Target encoding replaces a categorical group with its median resale price. The risk: if you compute the median on the full training set, each row's encoding contains information about its own label — this is data leakage, and it inflates cross-validation scores.

OOF encoding prevents this. The data is split into 5 folds. For each fold, only the other 4 folds are used to compute the median map. Each row's encoding is based only on rows it never saw during training.

def oof_group_median(group_series, y_series, n_splits=5, random_state=42):

Applied to: town → town_median_price, postal_sector → postal_sector_median_price, flat_model → flat_model_median_price.

Double OOF: These three encodings are also re-computed inside each fold of the stacking OOF loop. This means no leakage at either stage: not in the feature creation step, and not in the stacking step.

RandomizedSearchCV — Extra Trees (15 trials)

ExtraTrees has no learning rate, so the search space is simpler. The critical lesson: never include max_depth=None in the grid. Fully-grown trees on 150,634 rows can take 30+ minutes per fit. Capping at max_depth=30 brought ET search time to ~5 minutes.

Key Takeaways

  • Replacing 9,125-level ordinal-encoded postal with postal_sector_median_price (OOF median of 598 sectors) gave the model a real geographic price signal — the single biggest change in v9.
  • OOF target encoding must be recomputed inside the stacking OOF loop, not precomputed once — otherwise the encoded features leak the target into the validation fold.
  • LightGBM with num_leaves=300 and learning_rate=0.03 was the strongest individual model at val RMSE $21,541; the wider search grid was worth the extra compute.
  • ExtraTrees contributed a small but genuine signal (coefficient 0.13) — stacking still outperformed the best single model.
  • Capping max_depth for ExtraTrees (removing None) is essential — unbounded trees on 150K rows can run for 30+ minutes.
  • The OOF→Kaggle gap widened slightly (+$129 vs near-zero in v8), suggesting mild overfitting in postal_sector_median_price for rare sectors with few training rows.
