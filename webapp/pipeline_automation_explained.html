Why We Built This

By v17, we had 17+ experiment notebooks. Every new notebook started with a copy-paste of ~400 lines of identical boilerplate: loading data, engineering features, running the OOF stacking loop, blending with Ridge, and saving the CSV.

The problem: to test a new hyperparameter or a new OOF feature, you had to:

  1. Copy the entire notebook
  2. Find the right cell and edit a dict buried in the middle
  3. Hope you didn't accidentally change something else
  4. Re-run all 400+ lines of code

After v17, we extracted all the boilerplate into reusable src/ modules. Now each new experiment is a 3-cell notebook. You only touch Cell 1.

Architecture — What Lives Where

src/
  features/
    feature_engineering.py   ← engineer_features() + all constants
    oof_encoding.py           ← OOF group median target encoding
  models/
    stacker.py                ← 5-fold OOF stacking loop (4 base models)
    meta.py                   ← Ridge meta-learner with alpha sweep
  pipeline.py               ← run_pipeline(CONFIG) — orchestrates everything

notebooks/experiments/
  template_experiment.ipynb  ← copy this for every new experiment
  18_v16_pipeline_test.ipynb ← verified: reproduces v16 OOF $21,552 exactly

mlflow.db                   ← all experiment runs auto-logged here (project root)

The rule is simple: all reusable logic lives in src/. The notebook only holds your experiment-specific CONFIG dict. You never edit src/ files to run a new experiment — you only change the CONFIG.

The CONFIG Dict — Every Key Explained

The entire experiment is controlled by one Python dictionary. Here is the full CONFIG with every key annotated:

CONFIG = {
    "experiment_name": "v18",  # Label for MLflow. Change this every run.

    "data": {
        # Paths are relative to the notebook location (notebooks/experiments/)
        "train_path": "../../data/raw/train.csv",
        "test_path":  "../../data/raw/test.csv",
    },

    "features": {
        # OOF target encodings — each tuple adds one new column to X
        # Format: (group_column, new_feature_name, min_group_count)
        "encode_pairs": [
            ("town",           "town_median_price",           1),
            ("postal_sector",  "postal_sector_median_price",  1),
            ("flat_model",     "flat_model_median_price",     1),
            ("town_flat_type", "town_flat_type_median_price", 3),
            ("Tranc_Year",     "year_median_price",           1),
        ],
        # Extra columns to exclude from training (beyond the always-dropped set)
        "drop_cols": [],
    },

    "models": {
        # Remove a name to exclude that model from the stack entirely
        "active": ["xgb", "lgbm", "et", "catboost"],

        "xgb":      { ... },  # XGBRegressor params
        "lgbm":     { ... },  # LGBMRegressor params
        "et":       { ... },  # ExtraTreesRegressor params
        "catboost": { ... },  # CatBoostRegressor params
        "ridge_alphas": [0.001, 0.01, 0.1, 1.0, 10.0, 100.0],
    },

    "cv": {
        "n_splits": 5,      # number of cross-validation folds
        "random_state": 42,  # seed — keep fixed for reproducibility
    },

    "output": {
        # Always generated: OOF-averaged submission (honest score estimate)
        "submission_path": "../../outputs/submissions/sub_v18.csv",

        # Optional: uncomment to also generate a 100%-data retrain submission
        # "fulldata_submission_path": "../../outputs/submissions/sub_v18_fulldata.csv",

        "sample_path": "../../outputs/submissions/sample_sub_reg.csv",
    },
}

Running a New Experiment — Step by Step

1

Copy the template notebook

In Jupyter, right-click template_experiment.ipynb → Duplicate. Rename it with the next version number, e.g. 19_my_experiment.ipynb.

Never edit template_experiment.ipynb directly — it stays as the clean starting point.

2

Edit Cell 1 only

Change experiment_name to match your new notebook number. Then adjust whatever you want to test — hyperparameters, OOF features, or which models are active. See the sections below for specific examples.

Rule: Cell 1 is the only cell you ever touch. Cells 2 and 3 are boilerplate — do not edit them.

3

Run All Cells

In Jupyter: Kernel → Restart and Run All (or Shift+Enter through each cell). The pipeline will print progress as it runs — fold scores, Ridge alpha sweep, and final OOF RMSE.

Typical runtime: 20–40 minutes for the full 4-model stack on 150K rows.

4

Check results in Cell 3

Cell 3 prints the summary — OOF RMSE, Ridge weights, and per-model scores. The submission CSV is saved automatically to outputs/submissions/.

5

Compare in MLflow

Every run is automatically logged. Open the dashboard to compare this run against all previous experiments:

conda activate ml
cd "/Users/tehmenghai/SCTP-DSAI/Module 3/regression-challenge-hdb-price"
mlflow ui --backend-store-uri sqlite:///mlflow.db

Then open http://127.0.0.1:5000 in your browser.

What to Change — Common Experiments

Changing Hyperparameters

Find the model dict inside CONFIG["models"] and edit the values. Example: reduce XGB tree depth to test regularisation:

"xgb": {
    "n_estimators": 2000,
    "max_depth": 6,   # was 7 — testing shallower trees
    "learning_rate": 0.05,
    ...
}

Adding a New OOF Target Encoding Feature

OOF target encoding replaces a categorical column with the median resale price of that group — computed from the other 4 folds to prevent leakage. To test a new grouping, add a tuple to encode_pairs:

"encode_pairs": [
    ("town",           "town_median_price",           1),
    ("postal_sector",  "postal_sector_median_price",  1),
    ("flat_model",     "flat_model_median_price",     1),
    ("town_flat_type", "town_flat_type_median_price", 3),
    ("Tranc_Year",     "year_median_price",           1),
    # Add new encoding here:
    ("storey_range",   "storey_median_price",         5),
],

The tuple format is: (group_column, new_column_name, min_group_count)

  • group_column — any column that exists in train_fe after feature engineering (even if it's later dropped from X)
  • new_column_name — the name the new feature will have in the model's feature matrix
  • min_group_count — groups with fewer than this many observations use the global median instead. Use 1 for large groups (town), 3–5 for smaller groups

Leakage rule: The OOF encoding is recomputed inside each fold from training rows only. This means your new feature is always honest — no row influences its own encoding. This is handled automatically by the pipeline.

Removing an OOF Feature

Simply delete (or comment out) a tuple from encode_pairs. The pipeline will re-run without that column. Useful for ablation testing — finding out which encodings are actually contributing.

Excluding a Feature from Training

To drop a column that was added by feature engineering but you want to test without:

"drop_cols": ["street_freq"],  # test without street frequency encoding

The column is still computed internally (needed for group lookups) but excluded from the feature matrix X that the models train on.

Excluding a Base Model

Remove a name from "active" to drop it from the stack:

"active": ["xgb", "lgbm", "catboost"],  # test 3-model stack without ET

Ridge will re-learn the optimal weights among the remaining models automatically.

Two Submission Modes — OOF vs 100% Retrain

The pipeline always generates an OOF submission. You can optionally generate a 100% retrain submission on top of it.

Step 1 — 5-Fold OOF (always runs, ~20–40 min)
Each model trains on 80% of data per fold → predicts on the remaining 20%.
Ridge learns optimal blend weights from OOF predictions.
Saves: sub_v18.csv (test predictions averaged across 5 folds)
↓ optional
Step 2 — 100% Data Retrain (optional, +20–30 min)
All 4 base models retrain on all 150K rows.
Same Ridge weights as Step 1 — not retrained.
Saves: sub_v18_fulldata.csv (typically $50–100 better on Kaggle)
OOF SubmissionFull-Data Submission
Training data used80% per fold (5-fold CV)100% of 150K rows
Ridge weightsFitted on OOF predictionsSame weights as OOF step
Kaggle scoreHonest estimate, slightly higher RMSETypically $50–100 better
When to useDuring experimentation (fast iteration)For final Kaggle submission
How to enableAlways onUncomment fulldata_submission_path in CONFIG

Recommended workflow: Run OOF-only while iterating. When you find a config that beats your current best OOF RMSE, enable fulldata_submission_path, re-run the full notebook, and submit the _fulldata.csv file to Kaggle.

To enable the 100% retrain, uncomment one line in CONFIG:

"output": {
    "submission_path":          "../../outputs/submissions/sub_v18.csv",
    "fulldata_submission_path": "../../outputs/submissions/sub_v18_fulldata.csv",

MLflow — Tracking All Experiments

Every time you run the pipeline, MLflow automatically logs:

  • All hyperparameters for every model
  • OOF RMSE for each base model (XGB, LGBM, ET, CatBoost)
  • Final blended OOF RMSE
  • Ridge coefficient (weight) for each model
  • Best Ridge alpha
  • Which OOF encoding pairs were used

Full Pipeline Flow

This is exactly what happens inside run_pipeline(CONFIG) when you run Cell 2:

1. Load Data
Reads train.csv (150,634 rows) and test.csv (16,737 rows)
↓
2. Feature Engineering
Calls engineer_features() — adds 23 columns: distances, cyclical months, interactions, lease features, storey ranks, postal sector, block number
↓
3. Global OOF Encodings
Adds OOF target-encoded price columns from encode_pairs. Train: cross-validated (no leakage). Test: full-training-set maps.
↓
4. Build Feature Matrix
Drops identifier columns, target, raw high-cardinality cols, and any drop_cols. Aligns X_test columns to match X.
↓
5. 5-Fold OOF Stacking
Per fold: recomputes encode_pairs from training rows → trains all active models → predicts OOF on validation rows and on test set. Prints per-fold RMSE table.
↓
6. Ridge Meta-Blend
Sweeps 6 alpha values → fits Ridge on OOF predictions → learns optimal blend weights. Prints alpha sweep table with coefficients.
↓
7. Save OOF Submission
Saves submission_path CSV (test predictions averaged across 5 folds)
↓ if fulldata_submission_path set
7b. 100% Data Retrain (optional)
Retrains all base models on full 150K rows with full-data group maps → passes through existing Ridge weights → saves fulldata_submission_path CSV
↓
8. MLflow Logging
Logs all hyperparameters, OOF scores, Ridge weights, and best alpha to mlflow.db
