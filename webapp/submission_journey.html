3. CV vs Kaggle Gap Trend

The gap between CV/OOF RMSE and Kaggle RMSE measures how well the model generalises. A smaller gap = the CV score is a reliable predictor of Kaggle performance.

VersionCV/OOF RMSEKaggle RMSEGapWhat drove the change
v5$21,570$22,428$858Target leakage — town_median_price computed on full train
v6$21,818$22,124$306OOF encoding for town_median_price eliminated leakage
v7$21,841$21,906$65OOF stacking + fold-averaged test predictions
v8$21,708$21,805$973-model stack; slight gap increase (expected with added model)
Rule of thumb: Gap < $300 = well-calibrated. Gap < $100 = excellent. Gaps above $500 usually signal target leakage or overfitting to training data.

Log Target + OOF Encoding + Interaction Features
2026-04-22 • 05_advanced_tuning.ipynb • submitted

What we tried

  • log1p(resale_price) as training target
  • OOF town_median_price encoding (per-fold, prevents leakage)
  • 3 interaction features: dist_x_storey, lease_x_area, log_dist_to_cbd
  • Re-tuned XGB and LGBM with log target

Why

Log transform reduces the disproportionate RMSE penalty on expensive flats. OOF encoding directly fixes the $858 gap from v5.

3-Model Stack — Ridge Meta on XGB+LGBM+ET OOF
2026-04-23 • 07_extra_trees_stack.ipynb • 🏆 current best

What we tried

  • Added ExtraTreesRegressor as 3rd base model (in syllabus — bagging)
  • Ridge meta on [xgb_oof, lgbm_oof, et_oof] — 3 meta-features
  • Tuned ET: n_est=700, max_feat=0.8, depth=None, min_split=10

Learned: 0.45×XGB + 0.33×LGBM + 0.22×ET – 0.033

Why

ET (bagging) makes fundamentally different errors than XGB/LGBM (boosting). Adding a structurally different model gives Ridge more signal to correct systematic boosting errors.
